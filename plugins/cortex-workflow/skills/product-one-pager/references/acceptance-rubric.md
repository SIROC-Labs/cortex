# Acceptance rubric

Workflow step 7: verify the draft against every criterion before writing the file. Fix any failure and re-check.

- [ ] **1. House template structure.** The Owner lead field plus the six canonical sections, in order, on both entry paths — non-canonical input headings remapped, no invented sections. The experiment carve-out applied when the input is experiment/parity-like. Risks present only if proposed in Q&A and accepted (or carried from the reviewed input), between sections 4 and 5.

- [ ] **2. Provenance on every number.** MCP-verified from the right project (with a provenance phrase), owner-confirmed, or an owner-accepted inference with its justification in plain prose. No fabricated benchmarks.

- [ ] **3. Clean output.** No `[Proposed]` or `[Assumed]` anywhere in the file — headings or body; inference justifications kept as prose; a tagged input emitted untagged.

- [ ] **4. Q&A ran one decision per turn, before drafting.** Every load-bearing gap was put to the owner and resolved into an owner decision; no batched findings; no untouched placeholder carried into the output.

- [ ] **5. Gaps paired with concrete proposals.** Each flagged gap resolved into a proposed line or section (or a deliberate, surfaced placeholder) — not just a list of what's missing. Weak evidence down-weighted or replaced, not presented as fact.

- [ ] **6. Subtractive cuts made when warranted.** If the input contained unsubstantiated claims, redundant qualifiers, or non-load-bearing content, at least one such item was actually removed.

- [ ] **7. Proportionate to the input.** An experiment/parity hunch produced a tight brief; a richer feature input produced a fuller one.

- [ ] **8. Output file correct.** Written to `./<slug>.one-pager.md` in the cwd; no existing file overwritten.

- [ ] **9. Key user stories well-formed per template §3.** Human actors (an end user or an internal person — no system, app, service, component, API, or SDK), all three components (user + situation, motivation, outcome), high-level, 1–5 stories — unless carved out. Defective input stories were challenged in Q&A and re-anchored on the human or relocated out of §3, not passed through; valid internal-person and end-user actors kept.

- [ ] **10. Completeness gate passed.** Owner, a named human user + job-to-be-done, at least one piece of grounded evidence, at least one measurable success criterion, and (unless carved out) well-formed Key user stories — each a recorded owner decision; nothing silently omitted or invented to fill a hole.

- [ ] **11. Analytics MCP matched to the product.** Right-project / wrong-project / not-connected / unavailable distinguished; no query ran against a wrong-product project and no other product's numbers were attributed to this one; every non-verified state triggered an owner choice plus manual Evidence + success input.

- [ ] **12. Every in-scope section asked, not assumed** — explicitly including Out of scope and Launch timeline, with owner decisions on both the rollout approach and the release timing (a date/period or an explicit "to be decided"). Every Q&A turn offered an explicit "write your own" option; set-shaped turns (Evidence, success metrics, Risks, story selection) allowed multiple selections.

- [ ] **13. Authorship, not transcription.** The output names a specific problem, grounds evidence, and proposes KPI targets and risks — it does not merely reformat the source.
